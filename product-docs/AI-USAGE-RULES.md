# AI 使用規則｜省額度操作守則

本文件用來規範：什麼情況需要問 ChatGPT，什麼情況應該先看文件、自己判斷或用 Terminal 確認。

目的不是少用 AI，而是把 AI 用在最有價值的地方。

---

## 一、核心原則

凡是問過 2 次以上的問題，都要整理成 SOP、模板或檢查表。

ChatGPT 不應該當成臨時記憶。

正確分工：

| 工具 | 主要用途 |
|---|---|
| ChatGPT | 判斷、設計、排錯、商業策略、安全風險 |
| Terminal | 執行指令、確認狀態 |
| 文件 | 保存固定流程、話術、SOP |
| Cursor | 寫檔案、改程式 |
| Codex | 審查大型程式、架構、安全性 |

---

## 二、不需要問 AI 的情況

以下情況可以自己判斷，不需要每次問 ChatGPT。

### 1. Git 狀態乾淨

如果 Terminal 顯示：

- nothing to commit, working tree clean

代表目前沒有未儲存修改，狀態正常。

不用再問：

- 這樣有成功嗎？
- 這樣可以嗎？
- 這樣會不會壞掉？

---

### 2. 已經固定的專案規則

以下規則已固定，不需要重複詢問：

- 商品母版不要直接改
- 新客戶要複製母版
- 客戶版才可以改店名、服務、顏色、價格
- 不要刪 backend / customer-ui / owner-admin / docs / scripts
- 不要把 Token、密碼、驗證碼放進 GitHub
- 不要請客戶提供密碼或驗證碼

---

### 3. 只是查檔案或看文件

如果只是要看既有說明，先打開文件，不要先問 AI。

常用文件：

| 文件 | 用途 |
|---|---|
| PRODUCT-MASTER.md | 商品母版定位 |
| PROJECT-FOLDER-MAP.md | 專案資料夾用途 |
| product-docs/CLIENT-INFO-CHECKLIST.md | 客戶資料蒐集 |
| product-docs/PRICING-DRAFT.md | 報價草稿 |
| product-docs/CLIENT-DELIVERY-CHECKLIST.md | 交付檢查 |
| product-docs/CLIENT-LINE-SETUP-FLOW.md | LINE 官方帳號設定流程 |
| product-docs/AI-USAGE-RULES.md | AI 使用規則 |

---

## 三、需要問 AI 的情況

以下情況適合問 ChatGPT。

### 1. 出現錯誤訊息

例如：

- error
- failed
- not found
- permission denied
- fatal
- 找不到
- 拒絕存取
- 部署失敗

這種情況要貼完整錯誤訊息給 ChatGPT。

---

### 2. 需要商業判斷

例如：

- 這個功能值不值得做？
- 這個方案要賣多少？
- 買斷和租用怎麼分？
- 客戶要求這樣合不合理？
- 是否要收維護費？

這類問題適合問 AI。

---

### 3. 需要系統架構判斷

例如：

- 要用 Notion 還是資料庫？
- 要用 Cloudflare 還是其他平台？
- LINE OA 要怎麼接？
- 這個功能要放前端還是後端？
- 是否有安全風險？

這類問題適合問 AI。

---

### 4. 需要寫正式文件或模板

例如：

- 客戶成交 SOP
- LINE 官方帳號話術
- 報價單內容
- 交付說明
- 維護合約條款

第一次可以請 AI 寫，之後固定成文件保存。

---

## 四、問 AI 前的檢查

問 ChatGPT 前，先問自己 4 個問題：

1. 這個問題之前問過嗎？
2. 文件裡有沒有答案？
3. Terminal 有沒有明確錯誤？
4. 這是固定流程，還是新的判斷？

如果是固定流程，先看文件。

如果是新判斷、錯誤、商業策略或安全問題，再問 AI。

---

## 五、省額度工作流程

正確流程：

| 次數 | 做法 |
|---|---|
| 第一次遇到問題 | 可以問 ChatGPT |
| 第二次遇到類似問題 | 請 ChatGPT 整理成文件 |
| 第三次以後 | 直接看文件，不再重複問 |

---

## 六、專案操作時的固定判斷

### Git 狀態

| 顯示 | 意思 | 是否需要問 AI |
|---|---|---|
| nothing to commit, working tree clean | 狀態乾淨 | 不需要 |
| Changes not staged | 有檔案改了但尚未加入 | 可以先照 SOP |
| untracked files | 有新檔案尚未追蹤 | 視情況問 |
| error / failed / fatal | 有錯誤 | 需要問 |

---

### 商品母版判斷

| 情況 | 正確做法 |
|---|---|
| 修改通用文件 | 可以改母版 |
| 修改客戶店名 | 不改母版，改客戶版 |
| 新客戶成交 | 複製母版 |
| 測試新功能 | 先分支或另建測試版 |
| 客戶要求客製 | 在客戶版處理 |

---

## 七、最容易浪費額度的行為

應盡量減少：

- 每一步都問有沒有成功
- 同一件事重複確認很多次
- 已經有文件還重新問
- 小修改也要求 AI 產生完整大文件
- 反覆要求生圖或修圖但沒有先定稿
- 在同一個長對話中混合太多不同任務

---

## 八、最值得使用 AI 的地方

以下用途值得使用高階 AI：

- 系統架構設計
- 安全風險檢查
- 客戶流程設計
- 商業模式判斷
- 報價策略
- 合約與權限風險
- 錯誤排查
- 功能是否值得開發
- 商品化文件整理

---

## 九、本專案建議使用方式

對 beauty-studio-booking 專案，建議這樣分工：

- 文件查詢：先看 product-docs
- 狀態確認：先看 git status
- 錯誤排查：問 ChatGPT
- 程式修改：用 Cursor
- 版本保存：用 Terminal + GitHub
- 大型審查：用 Codex
- 商業判斷：問 ChatGPT

---

## 十、一句話規則

固定流程交給文件。

執行結果交給 Terminal。

新的判斷才交給 ChatGPT。
