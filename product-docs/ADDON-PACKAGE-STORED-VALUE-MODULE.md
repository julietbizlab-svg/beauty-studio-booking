# 加購模組規劃 v1｜包卡／儲值／堂數

> **版本**：加購模組規劃 **v1**（文件規劃，**不實作程式**）  
> **用途**：說明加購要賣什麼、適合哪條產品線、業主／客人畫面、資料與 API 之後怎麼做。  
> **讀者**：老闆、業務（少寫工程詞；工程細節另開任務包）。  
> **母版**：建在「基礎款 v1.0」之上，**另報價**。  
> **相關**：產品線見 [PRODUCT-LINE-COMPARISON.md](PRODUCT-LINE-COMPARISON.md)；基礎斷點見 [BASELINE-V1-SNAPSHOT.md](BASELINE-V1-SNAPSHOT.md)

**一句話**：客人先買方案（包卡／儲值／學期卡），業主在手機上**人工開卡、扣堂、扣金額**；系統幫你記剩餘，**不幫你收錢**。

---

## 0. 接案先分產品線

| 產品線 | 適合誰 | 加購常需要什麼 |
|--------|--------|----------------|
| **① 美業個人工作室版** | 美甲、美睫、紋繡、美髮、美容（一對一） | 儲值金、包卡、剩餘堂數、已開卡／未開卡 |
| **② 小型教室版** | 瑜伽、小班舞團、小班健身 | 堂數卡、學期卡、扣堂、剩餘堂數、已開卡／未開卡 |

> ⚠️ 兩條產品線的**基礎預約邏輯不同**（一對一 vs 多人名額）。  
> 加購「記堂數／儲值」概念可類似，但**教室版必須等基礎教室產品完成後**再接，不能硬塞進現有美業一對一系統。

### 與基礎款的界線

| 基礎款 v1.0（已完成，已報價內） | 本加購 v1（另報） |
|----------------------------------|-------------------|
| LINE 預約、月曆選日、長時不重疊 | 包卡／堂數卡／學期卡 |
| 業主月曆查預約、服務／時段／店面 | 儲值金、剩餘堂數 |
| owner 後端驗證 | 已開卡／未開卡、扣堂／扣款紀錄 |

## 1. 產品線①｜美業個人工作室版加購

### 1.1 適合業態

美甲、美睫、紋繡、美髮、臉部護理、個人美容工作室。

### 1.2 建議賣什麼（可拆賣）

| 銷售包 | 給客戶一句話 |
|--------|--------------|
| 包卡 | 買 N 堂護理／美甲，系統記剩幾堂 |
| 儲值金 | 先存一筆錢，消費時業主手動扣金額 |
| 包卡＋儲值 | 兩種都要 |
| 維護代管 | 上線後幫你登記、查帳、小調整（併月維護） |

### 1.3 業主端需要看到什麼

| 項目 | 說明 |
|------|------|
| 客人姓名／LINE | 認得出是誰 |
| 方案名稱 | 例：臉部 10 堂、儲值 5000 |
| 總堂數／已用／剩餘堂數 | 包卡用 |
| 儲值總額／已用／剩餘金額 | 儲值用 |
| 狀態 | **未開卡／已開卡／已用完／已停用** |
| 開卡日、到期日 | 選填 |
| 扣堂紀錄、扣款紀錄 | 何時扣、扣多少、備註 |
| 備註 | 贈送、轉讓等 |

**業主要能做的事（第一版）**：新增方案、開卡、手動扣堂、手動扣款、停用、查紀錄。

### 1.4 客人端需要看到什麼（唯讀）

| 項目 | 說明 |
|------|------|
| 我的包卡 | 方案名稱 |
| 剩餘堂數 | 還能用幾堂 |
| 開卡狀態 | 未開／已開／用完／停用 |
| 到期日 | 有設才顯示 |
| 儲值餘額 | 剩多少錢 |
| 使用紀錄 | 最近扣堂／扣款摘要 |

客人**不能**自己改剩餘堂數或儲值。

## 2. 產品線②｜小型教室版加購

### 2.1 適合業態

瑜伽、小班舞團、小班健身、小型課程教室。

### 2.2 前提（重要）

- 教室版**基礎功能**（班表、名額、報名名單、滿班）須先做完。  
- 本加購掛在「有學員、有堂數可扣」之後，不是美業基礎款的直接外掛。

### 2.3 建議賣什麼（可拆賣）

| 銷售包 | 給客戶一句話 |
|--------|--------------|
| 堂數卡 | 買 N 堂課，每次上課扣 1 堂 |
| 學期卡 | 一整期／一季方案，記剩餘堂數與有效期 |
| 扣堂＋剩餘 | 老師上課後手動扣堂，學員看剩餘 |
| 已開卡／未開卡 | 付款後未開課前算未開卡；正式啟用後已開卡 |
| （延伸）儲值金 | 選配，與美業類似但綁「課」而非「一對一療程」 |

報價時也可另談：候補、簽到（見 §6 先不做清單；可列第二階段）。

### 2.4 業主／老師端需要看到什麼

| 項目 | 說明 |
|------|------|
| 學員姓名／LINE | 認人 |
| 方案類型 | 堂數卡／學期卡 |
| 總堂數／已用／剩餘 | |
| 狀態 | **未開卡／已開卡／已用完／已停用／已過期** |
| 學期／有效區間 | 學期卡用 |
| 扣堂紀錄 | 哪一堂課、何時扣、誰操作 |
| 備註 | |

**要能做的事（第一版加購）**：登錄方案、開卡、手動扣堂、停用、查剩餘與紀錄。  
（報名名單、滿班屬**教室基礎款**，不屬於本加購文件範圍。）

### 2.5 學員端需要看到什麼（唯讀）

| 項目 | 說明 |
|------|------|
| 我的堂數卡／學期卡 | 方案名稱 |
| 剩餘堂數 | |
| 開卡狀態 | |
| 到期日／學期結束 | |
| 扣堂／使用紀錄 | 摘要 |

## 3. Notion 需要新增哪些資料表（規劃，不建庫）

> 以下**只規劃欄位**，本階段**不建立** Notion、不改程式、不改現有四表。

兩條產品線可共用「方案／紀錄」概念；實作時可用同一組表＋「產品線／方案類型」欄位區分，或分兩套表（工程再定）。

### 3.1 `package_cards`（包卡／堂數卡／學期卡）

| 建議欄位 | 說明 |
|----------|------|
| 方案名稱 | 例：臉部 10 堂、瑜伽学期 20 堂 |
| 產品線 | 美業／教室（選項） |
| 方案類型 | 包卡、堂數卡、學期卡（選項） |
| LINE userId、客人／學員姓名 | |
| 總堂數、已使用堂數、剩餘堂數 | |
| 狀態 | 未開卡、已開卡、已用完、已停用（教室可加：已過期） |
| 開卡日、到期日 | |
| 備註、建立時間 | |

### 3.2 `stored_value_accounts`（儲值帳戶｜美業為主，教室選配）

| 建議欄位 | 說明 |
|----------|------|
| 帳戶名稱、LINE userId、姓名 | |
| 儲值總額、已使用金額、剩餘金額 | |
| 狀態 | 啟用、已用完、已停用 |
| 產品線、備註、建立時間 | |

### 3.3 `usage_logs`（扣堂／扣款／開卡紀錄）

| 建議欄位 | 說明 |
|----------|------|
| 紀錄標題、LINE userId、姓名 | |
| 類型 | 扣堂、扣款、開卡、加值、調整 |
| 關聯方案 ID | 對應 package 或儲值 page |
| 數量／金額 | 扣幾堂或多少元 |
| 操作者 | 業主／老師（後端寫入） |
| 發生時間、備註 | |
| 關聯預約／課程 ID | 選填；第一版可空 |

## 4. API 需要新增哪些路徑（規格，不實作）

寫入一律 **owner／老師後端驗證**；客人／學員僅唯讀自己的資料。

### 4.1 業主端（Owner）

| 方法 | 路徑（建議） | 用途 |
|------|--------------|------|
| GET | `/api/owner/packages` | 方案列表（可篩客人、狀態、產品線） |
| POST | `/api/owner/packages` | 新增包卡／堂數卡／學期卡 |
| PATCH | `/api/owner/packages/:id` | 開卡、停用、改到期日、備註 |
| POST | `/api/owner/packages/:id/deduct` | 手動扣堂 |
| GET | `/api/owner/stored-value` | 儲值列表 |
| POST | `/api/owner/stored-value` | 登記儲值 |
| PATCH | `/api/owner/stored-value/:id` | 停用、備註 |
| POST | `/api/owner/stored-value/:id/deduct` | 手動扣款 |
| GET | `/api/owner/usage-logs` | 扣堂／扣款紀錄 |

### 4.2 客人／學員端（唯讀）

| 方法 | 路徑（建議） | 用途 |
|------|--------------|------|
| GET | `/api/me/packages` | 我的包卡／堂數卡／學期卡 |
| GET | `/api/me/stored-value` | 我的儲值 |
| GET | `/api/me/usage-logs` | 我的使用紀錄 |

須確保只能查自己的身分，不可偷看他人。

## 5. 第一版加購可以先做什麼

建議**先做美業個人工作室版**（有現成基礎款母版），教室版等基礎教室產品再排：

| 優先 | 內容 | 產品線 |
|------|------|--------|
| P0 | 業主手動新增包卡、開卡、扣堂、查剩餘與扣堂紀錄 | 美業 |
| P0 | 客人端唯讀「我的包卡／剩餘／狀態」 | 美業 |
| P1 | 儲值金登記、手動扣款、客人看餘額 | 美業 |
| P2 | 同上能力掛到教室版（堂數卡／學期卡） | 教室（需先有教室基礎） |

**第一版原則**：人工操作、不含金流、不含自動扣、畫面簡單能在手機完成。

### 銷售包建議（老闆對外）

| 包 | 建議 |
|----|------|
| A 僅包卡／堂數卡 | 加購單項 |
| B 僅儲值金 | 加購單項 |
| C 包卡＋儲值 | 打包 |
| D 維護代管 | 併月費 |

金額區間可寫進 [PRICING-PACKAGES.md](PRICING-PACKAGES.md)；本文件不鎖死數字。

## 6. 哪些先不要做

客戶提起時請**另報**或明確說「本加購 v1 不做」：

| 不做 | 原因 |
|------|------|
| 金流／線上收款 | 複雜、金流合規與對帳另案 |
| 預約成功自動扣堂／自動扣款 | 易出錯；第二階段再談 |
| 候補、排隊 | 教室進階 |
| 簽到打卡 | 教室進階 |
| 複雜營收報表、分析儀表板 | 另案 |
| 多員工分帳、佣金 | 另案 |
| SaaS 多租戶、點數商城、CRM | 超出本模組 |

## 8. 驗收勾選（老闆版｜開發完成後再用）

- [ ] 已確認產品線是美業或教室（或兩階段）
