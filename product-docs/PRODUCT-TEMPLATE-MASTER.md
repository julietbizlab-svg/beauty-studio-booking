# 商品母版總文件

> **用途**：這是 `beauty-studio-booking` 整套商品的「母版說明書」。  
> 每接一個新客戶，**複製這套做法**，但所有帳號、網址、密碼都要換成**該客戶自己的**。  
> **讀者**：接案老闆、業務窗口（非工程師）。

---

## 1. 商品名稱

**美業一人工作室 LINE 預約管理系統**

---

## 2. 商品一句話介紹

讓個人工作室老闆可以用 **LINE** 完成客人預約與後台管理——客人不用裝 App，老闆用手機就能看預約、改服務、改營業時間。

---

## 3. 適合對象

- 美甲師
- 美睫師
- 美容師、護膚工作室
- 紋繡師、霧眉師
- 按摩、SPA 個人工作室
- 其他「一人接客、以預約制」的小型服務業

**關鍵字**：一人工作室、LINE 經營、預約制、不想一直回私訊排時間。

---

## 4. 第一版包含功能

### 客人端（LINE 預約頁）

| 功能 | 說明 |
|------|------|
| 客人 LINE 登入 | 從官方帳號或 LIFF 進入，自動辨識身分 |
| 服務列表 | 顯示可預約服務、價格、時長 |
| 日期選擇 | 選擇預約日期 |
| 時段選擇 | 依營業時段與服務時長顯示可選時間 |
| 建立預約 | 送出後寫入預約紀錄 |
| 我的預約 | 查看自己的預約 |
| 取消預約 | 依規則取消（狀態更新） |

### 業主端（LINE 管理頁）

| 功能 | 說明 |
|------|------|
| 業主登入 | 僅限授權的業主 LINE 帳號 |
| 今日預約 | 查看指定日期的預約清單 |
| 服務管理 | 新增、修改、上架／下架服務 |
| 營業時段管理 | 設定每週哪幾天、幾點到幾點可預約 |
| 店面設定 | 店名、主色、公告文字 |

---

## 5. 第一版不包含

以下若客戶提出，需**另外報價**或說明現階段不做：

- 金流（線上收款、訂金）
- 多員工排班
- 營收報表、數據分析
- 多店、連鎖管理
- SaaS 多租戶（一個平台服務很多家店）
- 複雜會員系統（點數、等級、儲值）

---

## 6. 技術組成（白話版）

不用懂程式，只要知道「每一塊負責什麼」：

| 組件 | 白話說明 |
|------|----------|
| **LINE LIFF** | 客人和業主都用 LINE 開啟預約頁／管理頁，像開網頁一樣方便 |
| **GitHub Pages** | 放客人看得見的前端畫面（網址給 LINE 連過來） |
| **Cloudflare Workers** | 放 API（大腦），處理預約、權限、跟 Notion 溝通 |
| **Notion** | 放預約資料（服務、時段、預約紀錄、店面設定） |
| **GitHub** | 保存程式與文件版本，方便維護與交付 |
| **Cursor／Codex** | 協助開發、檢查、上線；有安全疑慮時由 Codex 審查 |

**資料流（簡化）**：客人按 LINE → 前端頁面 → API → Notion 存檔。

---

## 7. 每個新客戶都要複製並更換什麼

複製的是**整套做法與程式母版**，以下必須換成**該客戶專屬**：

| 項目 | 說明 |
|------|------|
| 客戶自己的 LINE 官方帳號 | 對外營運用，不與其他客戶共用 |
| 客戶自己的 LINE Login Channel | 在 LINE Developers 建立 |
| 客戶自己的 LIFF ID | 客人端、業主端各一（或依方案） |
| 客戶自己的 Notion 資料庫 | 四表：服務、時段、預約、設定 |
| 客戶自己的 Notion Token | 只放後端，不給客人看 |
| 客戶自己的 Cloudflare Secrets | 後端密碼，每客戶獨立上傳 |
| 客戶自己的 owner LINE userId | 誰能開業主管理頁 |
| 客戶自己的 API_BASE_URL | 該客戶的 Cloudflare Worker 網址 |
| 客戶自己的品牌資料 | 店名、主色、公告、服務、價格、營業時間 |

**口訣**：一個客戶 = 一套帳號 + 一份資料，不混用。

---

## 8. 絕對不能複製給客戶共用的東西

> ⚠️ **以下違反會有資安與資料混亂風險，務必遵守。**

| 禁止共用 | 原因 |
|----------|------|
| ❌ 不共用 Demo Token | 別人可能看到或改到客戶資料 |
| ❌ 不共用 Demo Notion | 預約會混在一起 |
| ❌ 不共用 Demo Cloudflare Secrets | 等於多家店共用同一把後門鑰匙 |
| ❌ 不共用開發者私人帳號當正式客戶系統 | 客戶日後無法自主、維護糾紛多 |

**Demo 只用來展示成交流程**；簽約後一律建**客戶專屬**環境。

## 9. 報價建議

以下為**建議區間**，實際依客製程度、帳號是否代管、教學與維護範圍調整。  
詳細方案見 [PRICING-PACKAGES.md](PRICING-PACKAGES.md)。

| 方案 | 適合誰 | 建議售價 | 備註 |
|------|--------|----------|------|
| **Demo 展示** | 還在觀望、想看畫面 | 免費或低價展示 | 非正式上線系統 |
| **基礎買斷版** | 確定要、可自己用業主端小改 | **NT$ 15,000～25,000**（一次性） | 含上線與驗收 |
| **建置加維護版** | 不想碰技術、要長期有人顧 | 建置 **NT$ 20,000～35,000**<br>＋ 月維護 **NT$ 1,000～3,000** | 含定期檢查與小幅調整額度 |

**提醒**：

- 客製越多、客戶越不想自己弄 → 價格往上調
- 清單外功能（金流、報表、多店）→ 另報，不要免費順便做

## 10. 接案流程（8 步）

| 步驟 | 老闆要做的事 |
|------|----------------|
| **1. 客戶諮詢** | 了解店型、痛點、是否適合本商品 |
| **2. 確認需求** | 對照「包含／不包含」，管理期待 |
| **3. 報價** | 選 Demo／買斷／維護版，口頭或書面報價 |
| **4. 收訂金** | 建議收訂金後再開工（比例自訂） |
| **5. 建立客戶專屬帳號與資料庫** | 收資料表、建 Notion／LINE（交工程執行） |
| **6. 部署系統** | 後端上線、前端網址、LIFF 設定（交工程執行） |
| **7. 客戶驗收** | 客人端＋業主端實機勾選（見下方第 11 節相關驗收文件） |
| **8. 收尾款與交付** | 交操作說明、說明維護方式、留驗收紀錄 |

詳細工程步驟：[CLIENT-DELIVERY-SOP.md](CLIENT-DELIVERY-SOP.md)  
老闆版總覽：[OWNER-SALES-DELIVERY-FLOW.md](OWNER-SALES-DELIVERY-FLOW.md)

## 12. 安全規則

> 🔒 **以下為鐵律，違反可能造成客戶資料外洩或系統被盜用。**

| 規則 | 說明 |
|------|------|
| **Token 不可貼到前端** | 客人看得見的網頁只能有公開網址，不能有密碼 |
| **`.dev.vars` 不可 commit** | 本機密碼檔不可上傳 GitHub |
| **owner 權限一定要後端驗證** | 不能只在網頁藏按鈕，必須伺服器擋非業主 |
| **有安全疑慮先停下來，交給 Codex 審查** | 例如：能不能 deploy、能不能 push、密碼要放哪——不要老闆自己猜 |

涉及 **token、secret、正式 deploy、推 main** 的動作，一律走工程＋Codex 流程。

## 13. 母版使用方式

### 這份文件怎麼用

1. **當商品說明書**  
   向客戶或合夥人介紹「我們在賣什麼、不賣什麼」。

2. **當接案檢查表**  
   新案開工前對一遍：對象適不適合、資料齊不齊、有沒有要加價的項目。

3. **當複製範本**  
   新客戶專案：複製 `beauty-studio-booking` 程式母版 → 換成該客戶的 LINE、Notion、Cloudflare、品牌資料。

### 複製時必記

```
✅ 可以複製：程式架構、文件流程、驗收方式、報價邏輯
❌ 不可複製共用：Token、Notion 庫、Secrets、Demo 網址、業主 userId
```
