## 0.1 兩種商品適用情境（老闆必讀）

目前母版要當成**兩條產品線**來看，再對客戶報價：

| 產品線 | 一句話 | 現況 |
|--------|--------|------|
| **美業個人工作室版** | 一個時段通常只服務一位客人 | **基礎款 v1.0 已完成**，可賣、可套版 |
| **小型教室版** | 同一堂課可多人報名，有名額上限 | **尚未完成**；可沿用部分架構，但要另開任務包改資料與流程 |

### 一、美業個人工作室版（一人公司）

**適合**

- 美甲、紋繡、美髮、美睫  
- 臉部護理、個人美容工作室  

**核心規則（跟客人怎麼講）**

- 一個時段通常只服務一位客人  
- 服務有不同時長（30／60／120／180 分等）  
- 長時間服務會鎖住連續時段，避免兩個人時間疊在一起  
- 業主用月曆看哪一天有人預約  

**基礎款功能（已有）**

- 客人 LINE 預約  
- 客人月曆選可預約日期（只顯示有空檔的日子）  
- 業主月曆查詢預約  
- 服務項目管理  
- 營業時段管理  
- 店面設定  

**可加購（另報，規劃中）**

- 包卡、儲值金、剩餘堂數  
- 已開卡／未開卡、扣堂紀錄  

### 二、小型教室版（多時段、多人報名）

**適合**

- 瑜伽、小班舞團、小班健身  
- 小型課程教室  

**核心規則（跟客人怎麼講）**

- **同一個時段可以多人報名**  
- 每堂課有名額上限，**滿班後不可再報名**  
- 業主或老師要看每堂課的報名名單  
- 與美業「一對一鎖時段」不同，**不能直接共用同一套預約邏輯**  

**基礎款功能建議（尚未實作，報價當另案）**

- 課程班表、課程名額  
- 報名名單、滿班關閉  
- 老師查看今日課程  

**可加購（另報）**

- 學期套票、堂數扣抵  
- 候補、簽到、儲值金  

### 三、重要提醒（販售時一定要說清楚）

1. **現在已完成、可立即套版賣的**：主要是 **美業個人工作室版**（基礎款 v1.0）。  
2. **小型教室版**：可沿用 LINE、網頁、雲端架構與交付流程，但**資料怎麼存、怎麼算名額、怎麼顯示名單**要獨立任務包，不能拿美業一對一硬改硬塞。  
3. **未來接案開頭先問**：客戶是「一對一服務」還是「多人課程」？答錯產品線，上線後一定翻車。  
4. 完整比較表見：[PRODUCT-LINE-COMPARISON.md](PRODUCT-LINE-COMPARISON.md)

### 美業個人工作室版（目前主賣）

**注意**：需求若屬此類，先估「教室版」建置，不要用美業母版直接承諾上線日。

## 4. 第一版包含功能

### 客人端（LINE 預約頁）

| 功能 | 說明 |
|------|------|
| 客人 LINE 登入 | 從官方帳號或 LIFF 進入，自動辨識身分 |
| 服務列表 | 顯示可預約服務、價格、時長 |
| **月曆選日期** | **基礎款**：選服務後顯示整月月曆；僅「有開放營業且仍有空位」的日期可點；其餘（未開放、已過期、已額滿、今日時段已過）灰色不可點 |
| 時段選擇 | 點可約日期後，顯示當日可預約**開始時間**（依服務時長；120／180 分須整段連續空檔） |
| 建立預約 | 送出後寫入預約紀錄；後端再次檢查與現有預約**不可重疊** |
| 我的預約 | 查看自己的預約 |
| 取消預約 | 依規則取消（狀態更新） |

> **預約三步驟（基礎款）**：選服務 → 月曆選日期 → 選時段 → 確認預約。  
> 客人**不能**點沒空位的日期；點日期後才會看到可預約時段。  
> **長時服務規則（基礎款）**：服務占用「開始～開始+時長」；與他人預約時間區間重疊則不可約；**首尾相接不算重疊**。

### 業主端（LINE 管理頁）

| 功能 | 說明 |
|------|------|
| 業主登入 | 僅限授權的業主 LINE 帳號 |
| **月曆預約查詢** | **基礎款**：以月曆查看各日預約；點日期顯示當日預約清單 |
| 服務管理 | 新增、修改、上架／下架服務 |
| 營業時段管理 | 設定每週哪幾天、幾點到幾點可預約 |
| 店面設定 | 店名、主色、公告文字 |

## 6. 技術組成（白話版）

不用懂程式，只要知道「每一塊負責什麼」：

| 組件 | 白話說明 |
|------|----------|
| **LINE LIFF** | 客人和業主都用 LINE 開啟預約頁／管理頁，像開網頁一樣方便 |
| **GitHub Pages** | 放客人看得見的前端畫面（網址給 LINE 連過來） |
| **Cloudflare Workers** | 放 API（大腦），處理預約、權限、跟 Notion 溝通 |
| **Notion** | 放預約資料（服務、時段、預約紀錄、店面設定） |
| **GitHub** | 保存程式與文件版本，方便維護與交付 |
| **Cursor／Codex** | 協助開發、檢查、上線；有安全疑慮時由 Codex 審查 |

**資料流（簡化）**：客人按 LINE → 前端頁面 → API → Notion 存檔。

## 7. 每個新客戶都要複製並更換什麼

複製的是**整套做法與程式母版**，以下必須換成**該客戶專屬**：

| 項目 | 說明 |
|------|------|
| 客戶自己的 LINE 官方帳號 | 對外營運用，不與其他客戶共用 |
| 客戶自己的 LINE Login Channel | 在 LINE Developers 建立 |
| 客戶自己的 LIFF ID | 客人端、業主端各一（或依方案） |
| 客戶自己的 Notion 資料庫 | 四表：服務、時段、預約、設定 |
| 客戶自己的 Notion Token | 只放後端，不給客人看 |
| 客戶自己的 Cloudflare Secrets | 後端密碼，每客戶獨立上傳 |
| 客戶自己的 owner LINE userId | 誰能開業主管理頁 |
| 客戶自己的 API_BASE_URL | 該客戶的 Cloudflare Worker 網址 |
| 客戶自己的品牌資料 | 店名、主色、公告、服務、價格、營業時間 |

**口訣**：一個客戶 = 一套帳號 + 一份資料，不混用。

## 8. 絕對不能複製給客戶共用的東西

> ⚠️ **以下違反會有資安與資料混亂風險，務必遵守。**

| 禁止共用 | 原因 |
|----------|------|
| ❌ 不共用 Demo Token | 別人可能看到或改到客戶資料 |
| ❌ 不共用 Demo Notion | 預約會混在一起 |
| ❌ 不共用 Demo Cloudflare Secrets | 等於多家店共用同一把後門鑰匙 |
| ❌ 不共用開發者私人帳號當正式客戶系統 | 客戶日後無法自主、維護糾紛多 |

**Demo 只用來展示成交流程**；簽約後一律建**客戶專屬**環境。

## 12. 安全規則

> 🔒 **以下為鐵律，違反可能造成客戶資料外洩或系統被盜用。**

| 規則 | 說明 |
|------|------|
| **Token 不可貼到前端** | 客人看得見的網頁只能有公開網址，不能有密碼 |
| **`.dev.vars` 不可 commit** | 本機密碼檔不可上傳 GitHub |
| **owner 權限一定要後端驗證** | 不能只在網頁藏按鈕，必須伺服器擋非業主 |
| **有安全疑慮先停下來，交給 Codex 審查** | 例如：能不能 deploy、能不能 push、密碼要放哪——不要老闆自己猜 |

涉及 **token、secret、正式 deploy、推 main** 的動作，一律走工程＋Codex 流程。

## 13. 母版使用方式

### 這份文件怎麼用

1. **當商品說明書**  
   向客戶或合夥人介紹「我們在賣什麼、不賣什麼」。

2. **當接案檢查表**  
   新案開工前對一遍：對象適不適合、資料齊不齊、有沒有要加價的項目。

3. **當複製範本**  
   新客戶專案：複製 `beauty-studio-booking` 程式母版 → 換成該客戶的 LINE、Notion、Cloudflare、品牌資料。

### 複製時必記

```
✅ 可以複製：程式架構、文件流程、驗收方式、報價邏輯
❌ 不可複製共用：Token、Notion 庫、Secrets、Demo 網址、業主 userId
```
